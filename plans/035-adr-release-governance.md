# ADR-035: Release Governance & Vulnerability Disclosure Policy

**Status:** Accepted
**Date:** 2026-05-17
**Supersedes:** none
**Related:** Plan 033 (Groups B, D, F, G), `.github/workflows/release.yml`, `CHANGELOG.md`, `VERSION`

---

## Context

The repository has shipped 150+ PRs and is feature-complete enough for a first cut, yet:

- `VERSION` reads `0.1.0` but **0 GitHub Releases** exist.
- `CHANGELOG.md` accumulates ≥ 30 entries under a single `[Unreleased]` heading with no `## [0.1.0]` cut.
- `release.yml` workflow exists but has never produced a release artifact or tag.
- No `SECURITY.md` — security researchers have no disclosure channel; GitHub's "Report a vulnerability" UI surfaces a generic 404.
- No `CODE_OF_CONDUCT.md` — standard OSS governance gap.
- No agent skill exists for **release management**, so version bumps, changelog hygiene, and tag conventions are ad-hoc.
- README lacks status signals (CI, coverage, CodeQL, release badges).

Without a release cadence, downstream consumers cannot pin a version, the CHANGELOG drifts, and security disclosures must be ad-hoc to maintainers via public issues — violating responsible-disclosure norms.

---

## Decision

### 1. Semantic versioning + Conventional Commits

The project already uses Conventional Commits (`type(scope): description`, ≤72 chars — AGENTS.md Tier 2 §2,4). We formalize:

- Version source of truth: `VERSION` file at repo root, mirrored in every `package.json`.
- Bump rules:
  - `fix(...)` → patch
  - `feat(...)` → minor (until 1.0.0; post-1.0.0 also minor)
  - `feat(...)!` or `BREAKING CHANGE:` footer → major

### 2. Release-drafter as single source of CHANGELOG

Adopt `release-drafter/release-drafter@v6` configured at `.github/release-drafter.yml` with categories matching commit prefixes:

```yaml
categories:
  - title: '🚀 Features'
    labels: [feat]
  - title: '🐛 Fixes'
    labels: [fix]
  - title: '🔒 Security'
    labels: [security]
  - title: '🧹 Chores'
    labels: [chore, build, ci, infra]
  - title: '📝 Docs'
    labels: [docs]
```

On every push to `main`, a draft release is updated. The `release.yml` workflow promotes a draft to a published tag when a `release:cut` label is added to a PR titled `chore(release): vX.Y.Z`.

### 3. CHANGELOG sync

`CHANGELOG.md` MUST contain a `## [X.Y.Z] - YYYY-MM-DD` section for every published release, generated by `./scripts/release/sync-changelog.sh` from the drafted release notes. The `[Unreleased]` section is reset on every cut.

### 4. SECURITY.md (mandatory)

Create `SECURITY.md` at repo root with:

- Supported versions table.
- Private disclosure channel: GitHub Security Advisories (https://github.com/d-oit/do-epub-studio/security/advisories/new) — **no email** (per `privacy-first` skill).
- Response SLA: triage within 5 business days; fix or mitigation plan within 30 days for HIGH+, 90 for MEDIUM.
- Scope: code in `apps/**` and `packages/**`. Out of scope: third-party deps (file to upstream), social engineering, DoS via legitimate API rate.
- Cross-link to ADR-034 (ReDoS) and `docs/security.md`.

### 5. CODE_OF_CONDUCT.md

Adopt Contributor Covenant 2.1 verbatim at repo root, with the project name substituted. Enforcement contact: GitHub Security Advisories (no PII per `privacy-first` skill).

### 6. New agent skill — `release-management`

Create `.agents/skills/release-management/SKILL.md` (per `skill-creator`, ≤250 lines):

- Activates on user requests like "cut a release", "bump version", "publish vX".
- Workflow: verify quality gate → bump `VERSION` + every `package.json` → run sync-changelog → open `chore(release): vX.Y.Z` PR → on merge, promote draft release → verify deployed artifact.
- Hard rule: NEVER tag `main` directly. NEVER manually edit CHANGELOG for a release section — only via the sync script.

### 7. README badges

Add to `README.md` (top of file):

```markdown
[![CI](https://github.com/d-oit/do-epub-studio/actions/workflows/ci.yml/badge.svg)](https://github.com/d-oit/do-epub-studio/actions/workflows/ci.yml)
[![CodeQL](https://github.com/d-oit/do-epub-studio/actions/workflows/codeql.yml/badge.svg)](https://github.com/d-oit/do-epub-studio/actions/workflows/codeql.yml)
[![Release](https://img.shields.io/github/v/release/d-oit/do-epub-studio?include_prereleases)](https://github.com/d-oit/do-epub-studio/releases)
[![License: MIT](https://img.shields.io/badge/License-MIT-yellow.svg)](LICENSE)
```

### 8. AGENTS.md additions

Append to AGENTS.md Tier 1:

> - **MUST follow the disclosure process in `SECURITY.md` for any vulnerability — never open a public issue first.**

Append to AGENTS.md Tier 2:

> - **Releases MUST be cut via the `release-management` skill — no manual tags, no direct CHANGELOG edits.**

### 9. First cut — `v0.1.0`

Immediately after this ADR lands, cut `v0.1.0`:

1. Collapse current `[Unreleased]` into `## [0.1.0] - 2026-05-17`.
2. Open `chore(release): v0.1.0` PR.
3. On merge, release-drafter publishes the tag + GitHub Release with auto-generated notes.

---

## Consequences

### Positive

- External consumers can pin a version.
- Security researchers have a private channel; SLA is auditable.
- CHANGELOG is generated, not curated — eliminates drift.
- Agent skill removes release-process tribal knowledge.
- Badges surface quality signals to first-time visitors.

### Negative

- Marginal overhead per release (≈ 5 min of human-in-the-loop on the release PR).
- Conventional Commit discipline becomes a hard merge gate (already partially enforced).

### Risks

- Release-drafter may miscategorize PRs lacking labels; mitigated by `actions/labeler` config keyed on commit prefix.
- A bot-published release could leak in-progress features if the `[Unreleased]` cut is skipped; mitigated by the `release-management` skill's mandatory dry-run step.

- `https://github.com/d-oit/do-epub-studio/releases` lists `v0.1.0`.
- `https://github.com/d-oit/do-epub-studio/security/policy` resolves to `SECURITY.md`.

- AGENTS.md compliance self-check includes the two new bullets.
- A dry-run of the `release-management` skill on a synthetic `v0.1.1` patch succeeds end-to-end.

## Implementation

The following governance files were created per ADR-035:

- `SECURITY.md` — Vulnerability disclosure policy (ADR-035 §4). Includes supported versions table, GitHub Security Advisories channel, response SLA, scope/out-of-scope, and cross-links to ADR-034 and `docs/security.md`.
- `CODE_OF_CONDUCT.md` — Contributor Covenant 2.1 at repo root. Enforcement contact via GitHub Security Advisories. No PII per `privacy-first` skill.
- `.github/release-drafter.yml` — Release-drafter v6 config (ADR-035 §2) with categories for feat, fix, security, chore/build/ci/infra, and docs. Version resolver defaults to patch with major/minor labels.

**Not yet implemented:** Items 7 (README badges), 8 (AGENTS.md additions), and 9 (v0.1.0 cut) are deferred to subsequent PRs.
