## Cloudflare Platform Assessment

### Current Architecture

| Service | Binding | Purpose | Best Practice |
|---------|---------|---------|---------------|
| Workers | `do-epub-studio` | API + static serving | Hono routes, Zod validation, traceId |
| R2 | `BOOKS_BUCKET` | Book storage | Signed URLs only, no direct exposure |
| D1 | `DB` | Primary database | Via `env.DB.prepare()`, migrations |
| Durable Objects | `RATE_LIMITER` | Rate limiting | Fail-open on error |
| Cache API | `caches.default` | Edge cache for catalog | Per-isolate, 60s max-age |

### Key Findings

- **R2 access is Worker-mediated** — all file access goes through `/api/files/:bookId/:remainder` with signed URL verification. No raw R2 URLs exposed to clients.
- **Trace IDs emitted before path-length guards** — observability middleware runs first in the Hono pipeline.
- **Security headers on every response** — CSP, HSTS, X-Content-Type-Options applied via middleware.
- **CSP preservation** — EPUB content served with separate carve-out CSP, not the global Worker CSP.
- **Edge cache limitation** — cross-isolate invalidation not yet implemented (documented as P3, tracked in `plans/129`).

### Turso Recommendation

**Keep Cloudflare-native services.** Turso should only be considered as a scoped database-layer option if:
- D1 local development proves painful for developers
- Branchable databases are needed for feature work
- libSQL parity is a hard requirement

Turso local development (via `turso dev`) is a viable alternative for local dev ergonomics, but switching the entire production stack is not recommended.
