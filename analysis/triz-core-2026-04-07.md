# TRIZ Core Analysis - EPUB Studio
**Date:** 2026-04-07
**System:** d.o. EPUB Studio - Production EPUB reading and editorial workspace with gated access, offline reading, comments, highlights, and secure distribution.

---

## System Architecture Summary

- **Frontend:** Browser-based EPUB reader with offline-first state
- **Backend:** Cloudflare Workers
- **Storage:** R2 for EPUB files (object storage), Turso for app state (permissions, progress, comments, highlights, audit logs)
- **Auth:** Email + password with session management
- **Sync Model:** Explicit, resilient sync between browser local state and cloud

---

## Technical Contradiction #1: Security vs Usability

### Analysis

| Aspect | Detail |
|--------|--------|
| **Improving** | Security (access control, session revocation, audit logging) |
| **Worsens** | Usability (friction in authentication, session expiry, permission checks) |
| **Why** | Strong password requirements + session expiry + permission validation on every request creates login friction. Users must re-authenticate frequently. Revoking sessions on permission changes means users lose access immediately. |
| **Reality** | Assumed: Security always worsens usability. Real: With proper UX (password managers, SSO, biometric), security can enhance usability via consistent trust signals. |

### Resolution Approaches
- Implement "Remember me" with secure token rotation
- Use short-lived access tokens + refresh tokens for seamless re-auth
- Add progressive security UI that doesn't block core reading flow

## Technical Contradiction #2: Offline Availability vs Access Control

### Analysis

| Aspect | Detail |
|--------|--------|
| **Improving** | Offline availability (browser-first state, explicit sync) |
| **Worsens** | Access control (can't verify permissions offline) |
| **Why** | Offline mode caches EPUB content and annotation state locally. Without server connectivity, cannot verify if user's permission grant was revoked, or if book access expired. Risk: users continue reading after permission removal. |
| **Reality** | Assumed: Offline requires trusting local state. Real: Hybrid approach possible - cache last-known permission state with TTL, implement "graceful degradation" that blocks access after sync window expires. |

### Resolution Approaches
- Cache permission state with bounded TTL (e.g., 24 hours)
- Implement "zombie access" detection - prompt sync on reconnection
- Store permission expiry timestamp in local cache, enforce locally

## Technical Contradiction #3: Performance vs Flexibility

### Analysis

| Aspect | Detail |
|--------|--------|
| **Improving** | Performance (prefetching, caching, signed URLs for direct R2 access) |
| **Worsens** | Flexibility (tightly coupled to R2 + Turso schema, fixed sync windows) |
| **Why** | Optimized R2 signed URLs require pre-determined paths. Turso queries are schema-bound. Offline sync window is fixed. Adding new annotation types or storage backends requires migration and code changes. |
| **Reality** | Assumed: Performance optimization requires rigid architecture. Real: Can use adapter pattern and abstraction layers to maintain performance while allowing flexibility. |

### Resolution Approaches
- Abstract storage layer with interface adapters
- Use feature flags for experimental sync strategies
- Store flexible JSON payloads in Turso for extensibility

## Technical Contradiction #4: Simplicity vs Feature Richness

### Analysis

| Aspect | Detail |
|--------|--------|
| **Improving** | Simplicity (minimal UI, core reading experience) |
| **Worsens** | Feature richness (highlights, comments, annotations, admin panel) |
| **Why** | Clean reading experience conflicts with annotation overlays, comment threads, and admin controls. Each feature adds UI complexity, state management, and sync logic. Users want focused reading but also editorial tools. |
| **Reality** | Assumed: Feature-rich apps must be complex. Real: Progressive disclosure and context-aware UI can maintain simplicity while enabling depth. |

### Resolution Approaches
- Use progressive disclosure - annotation tools appear on selection
- Implement modes: "Reading Mode" vs "Editorial Mode"
- Lazy-load advanced features only when needed

## Technical Contradiction #5: Local-First vs Consistency

| Aspect | Detail |
|--------|--------|
| **Improving** | Local-first (browser stores state, works without connectivity) |
| **Worsens** | Consistency (conflicts possible when offline edits pile up) |
| **Why** | Browser IndexedDB stores highlights, progress, comments. When offline, changes accumulate. On sync, conflicts emerge if another user edited same content or permissions changed. No atomic resolution. |
| **Reality** | Assumed: Local-first always creates consistency conflicts. Real: CRDTs or operation transforms can maintain consistency without central coordination. |

### Resolution Approaches
- Implement CRDT-based annotations for conflict-free merging
- Use "last-write-wins" for progress (acceptable for reading position)
- Add conflict resolution UI for comments/highlights on sync
